# config — bootstrap, venv, global config

> **Maintenance:** see `lingtai-tui-anatomy` (at `tui/internal/preset/skills/lingtai-tui-anatomy/SKILL.md`).

This package manages the TUI's bootstrap sequence — the steps that run before any agent launches. It owns the Python runtime venv, the lingtai CLI upgrade path, addon verification, and the global config files under `~/.lingtai-tui/`.

## Components

- **`tui/internal/config/venv.go:32-117`** — runtime environment marker call site for `~/.lingtai-tui/runtime/venv/.lingtai-env.json`. Marker schema/evaluation/stamping are owned by the installed Python runtime via `python -m lingtai.venv_resolve env-marker {check,stamp} --venv <path>`; Go only consumes the high-level `missing` / `match` / `mismatch` / `error` result. Delegation failures, invalid output, and checker errors are treated as `unknown` and never delete the managed venv.
- **`tui/internal/config/venv.go:169-190`** — `NeedsVenv`: returns true when the managed runtime venv is missing, cannot import `lingtai`, or the delegated marker check returns a confirmed mismatch. A working legacy venv without a marker is accepted and stamped best-effort after the import probe succeeds. Unknown marker checks do not by themselves force deletion/recreation. A working PyPI wheel may still need conversion to local editable/dev mode; that is handled by the separate, consent-gated `CheckUpgrade` / `UpgradePythonRuntime` path instead of recreating the whole venv.
- **`tui/internal/config/venv.go:174-262`** — `EnsureVenv`: creates the runtime venv at `~/.lingtai-tui/runtime/venv/`. Uses `uv` if available (can auto-download Python 3.13), falls back to system Python. If the delegated marker check proves the managed venv belongs to another environment, removes that stale managed venv before invoking `uv venv` / `python -m venv`; unknown/error marker states are left in place. Verifies Python 3.11+, installs `lingtai`, asks the kernel-owned marker command to stamp best-effort, and symlinks CLI into PATH. On a dev machine, it installs the discovered `lingtai-kernel` checkout editable rather than PyPI when creating a fresh runtime.
- **`tui/internal/config/devmode.go:64-88`** — `findDevCheckouts`: discovers local LingTai development workspaces only through the explicit `$LINGTAI_DEV_ROOT` contract. The env var points at a root that directly contains sibling `lingtai-kernel/` and `lingtai/` checkouts. A workspace is valid when `lingtai-kernel/pyproject.toml` exists and a sibling `lingtai/` TUI repo exists. Only `lingtai-kernel` is passed to `pip install -e`; the Go `lingtai/` repo is a workspace marker, not a Python package.
- **`tui/internal/config/venv.go:437-449`** — `CheckUpgrade`: thin wrapper over `UpgradePythonRuntime(globalDir, false, ...)` that returns `true` only when a real PyPI upgrade or dev-mode conversion was verified post-install. This is a **mutating** operation gated on explicit human consent for the current launch — it must only be invoked via `main.go`'s shared startup-update preflight (`maybeCheckAndPromptKernelUpgrade`, which prompts only after the read-only `InspectKernel` below reports an actionable update; runs for EVERY default interactive launch — existing project, first-run, and the no-project launcher alike, not only "returning users") or an explicitly-invoked path such as `doctor`/`self-update`/`/update`, never automatically. Any failure is silent so the TUI can still launch.
- **`tui/internal/config/kernel_update.go:39-122`** — `InspectKernel(globalDir) KernelStatus` (thin wrapper, `39-41`) over `inspectKernel` (`43-122`): **read-only** kernel version check — no pip/uv/brew command, ever. Reuses the same probes the mutating path uses (`pythonLingtaiVersion`, `isEditableLingtaiInstall`, `findDevCheckouts`, `fetchLatestPyPIVersion`) so the classification cannot drift from what `RunKernelUpdate` would actually do. `NeedsUpdate` is true only for a missing/broken venv (repair is itself the "update") or a real PyPI version behind latest; editable/dev-checkout-already-matching, and up-to-date all report `NeedsUpdate=false`. `main.go`'s `runVersionPreflight` calls this — via `maybeCheckAndPromptKernelUpgrade` — for EVERY default interactive launch, before the no-project decision gate, whether or not a project currently exists, interactive or not, before ever deciding whether to prompt. install.sh installs the kernel by default, so first-run is not exempted.
- **`tui/internal/config/kernel_update.go:151-210`** — `RunKernelUpdate(globalDir, force) DoctorReport` (thin wrapper, `151-153`) over `runKernelUpdate` (`155-210`): the mutating kernel-only apply step — venv repair (mirroring `checkPythonRuntime`) plus `UpgradePythonRuntime`, but never the TUI binary (no brew) or file-search sidecar. Shared by `main.go`'s consent-driven startup upgrade, the `/update` interactive command (`tui/internal/tui/update.go`), and available to a future `/doctor` gated heal. `RunDoctorUpdate` (below) is left independent and unchanged for existing `doctor`/`self-update` callers.
- **`tui/internal/config/venv.go:458-481`** — `RuntimeReadyError` / `RuntimeReady(globalDir) error`: the sole readiness-check surface for every caller that is NOT the interactive startup preflight. **Read-only** — delegates to `NeedsVenv` and never creates, repairs, or upgrades anything. Returns nil when the managed venv exists and `lingtai` is importable; otherwise returns an actionable `*RuntimeReadyError` so the caller can surface a clear message instead of silently installing/repairing. There is deliberately no automatic "ensure the runtime" function any more (`EnsureRuntime`/`EnsureRuntimeQuiet`/`ensureRuntimeWithOptions`/`RuntimeEnsureOptions` were removed): install.sh installs the kernel by default, and the ONLY place the TUI may automatically install or repair it is `main.go`'s shared interactive startup preflight (`runVersionPreflight` → `maybeCheckAndPromptKernelUpgrade` → consent-gated `RunKernelUpdate`). Every other caller — returning-user `prepareApp`, the first-run wizard, headless spawn, and the post-Create project-launch phase — calls `RuntimeReady` instead, so a declined or absent runtime is never silently repaired behind that decision.
- **`tui/internal/config/venv.go:578-892`** — `RunDoctorUpdate` runtime path: verifies the TUI binary against the latest GitHub release, reports the detected install method, routes forced TUI updates through `RunTUIUpdate`, repairs a missing/broken/confirmed-mismatched runtime venv, then runs `UpgradePythonRuntime(force=true)`, and asks the kernel-owned marker command to write/refresh after successful probes or upgrades. Marker check errors are surfaced as warnings but do not delete the managed venv. All command stdout/stderr is captured after command completion and surfaced; nothing is silently swallowed.
- **`tui/internal/config/venv.go:701-842`** — `detectTUIInstallMethod` and helpers: read `~/.lingtai-tui/install.json` source metadata (`lingtai.tui.install/v1`) and match it against the running executable (including the `lingtai` alias symlink in `bin_dir`), then fall back to Homebrew path/env detection (`HOMEBREW_PREFIX`, common Homebrew/Linuxbrew prefixes) or `unknown/other`. Source metadata wins over Homebrew-looking paths so source installs into a Homebrew bin dir are still reported as source.
- **`tui/internal/config/tui_updater.go:14-366`** — `TUIUpdater` interface plus `RunTUIUpdate` / `RunManualTUIUpdate` backend routing. The Homebrew backend runs the existing `brew upgrade lingtai-ai/lingtai/lingtai-tui` flow, with doctor/self-update optionally preceding `brew update` (`IncludeHomebrewUpdate`) and full-path command reporting (`ResolveHomebrewPath`). The source backend (`sourceTUIUpdater.Upgrade`) requires a known release tag, reads source metadata from `install.json`, runs the versioned `install.sh --update --prefix <prefix> --version <tag> --non-interactive` contract, then verifies the new binary version, refreshed metadata, and runtime venv health. Unknown installs remain explicit unsupported command results instead of running brew. Callers keep version checks/prompting; the backend owns only the mutation-or-guidance for one install method.
- **`tui/internal/config/venv.go:1222-1391`** — `UpgradePythonRuntime`: checks the installed Python `lingtai` runtime. If a local dev checkout is discovered and the runtime is not already editable for that checkout, it runs editable install (`uv pip install -e <lingtai-kernel> -p <venv>` or venv `pip install -e <lingtai-kernel>`) before any PyPI comparison. Next, a **bundle-provenance gate** (`kernelBundleProvenance`, reads `install.json`'s additive `kernel_source`/`kernel_bundle_id`/`kernel_version`/`kernel_provider` fields written by `install.sh`'s mandatory bundle path — see `RELEASING.md` "Source selection") applies to **both routine and forced** checks when the runtime was installed from a pinned release-bundle artifact: a routine check silently skips (same as before), and `force=true` (doctor / `/update --force`) reports that the kernel is pinned to the compatible bundle and points at the bundle-update path instead of running any PyPI query or install — "force" means "run the check even if we'd otherwise consider it up to date," not "abandon the pin and install an arbitrary PyPI version." Only metadata with no `kernel_source` at all (installs that predate the bundle path) reaches the pre-existing legacy PyPI-compare/upgrade path unchanged — that path is intentionally left in place for migration continuity, not as the product's canonical install source going forward (every new install goes through `install.sh`'s mandatory bundle path, which never reaches PyPI for LingTai itself). Otherwise it preserves existing editable installs, compares to PyPI, runs upgrade if needed, verifies post-install import/version, and writes/refreshes the runtime environment marker best-effort after successful verified states.
- **`tui/internal/config/venv.go:483-561`** — `DoctorOptions` / `CommandRunner` / `TUIInstallMethod` / `TUIInstallInfo` / `tuiInstallMetadata`: dependency-injection seams so tests stub brew/pip/python/home/env without dialing the network, plus the small install-method result type used by doctor reporting and updater routing. `tuiInstallMetadata` carries `repo_url`, `requested_ref`, `resolved_ref`, `resolved_commit`, `stamped_version` so the source updater can drive and re-verify `install.sh --update`. Production callers leave the seams nil and get real `exec.Command` + `http.Client` + `os.UserHomeDir` / `os.LookupEnv`.
- **`tui/internal/config/venv.go:894-1001`** — `checkFileSearchNative` / `FileSearchStatus` / `timeoutCommandRunner` / `FileSearchNativeStatus`: asks the managed Python runtime which file-search backend it will use (`RustFileIOBackend` sidecar vs Python fallback) and reports both packaged sidecar status and local Cargo availability for `/doctor` / startup guidance. The probe runs through a `timeoutCommandRunner` (3s default) so startup and `/doctor` cannot hang on a slow or broken managed Python runtime. A `ModuleNotFoundError` for `lingtai.services.file_io_sidecar` (a runtime that predates the Rust sidecar release) is reported as `FileSearchStatus{Unsupported: true}` rather than an error; `checkFileSearchNative` then emits a single `DoctorInfo` line and the report stays healthy.
- **`tui/internal/config/venv.go:365-399`** — `EnsureAddons`: reads `init.json`'s `addons` map, verifies each addon is importable as `lingtai.addons.<name>`. Error surfaces which addon is missing and suggests `pip install --upgrade lingtai`.
- **`tui/internal/config/venv.go:340-363`** — `CheckTUIUpgrade`: compares running TUI binary version against latest GitHub release. Returns tag if upgrade available. Used by the startup-time prompt in `main.go`.
- **`tui/internal/config/venv.go:1059-1148`** — `CompareReleaseVersions` / `ReleaseComparison`: classifies strict `vX.Y.Z` release comparisons as update-available, up-to-date, current missing, dev build, current non-semver, or latest non-semver; `/doctor` uses this so hash-like current versions do not look up-to-date.
- **`tui/internal/config/venv.go:268-290`** — `linkLingtaiCLI`: symlinks `lingtai` CLI from venv into brew prefix or `~/.local/bin`.
- **`tui/internal/config/global.go:31-42`** — `Config`: global config at `~/.lingtai-tui/config.json`. Keys map env-var names to API key values.
- **`tui/internal/config/global.go:79-141`** — `TUIConfig`, `AllowedMailPageSizes`, and `NormalizeMailPageSize`: TUI preferences at `~/.lingtai-tui/tui_config.json`; `mail_page_size` accepts exactly 100/200/500/1000/2000 and every unsupported value normalizes to the finite default 200.
- **`tui/internal/config/global.go:395-442`** — `WriteEnvFile`: writes `~/.lingtai-tui/.env` from Config.Keys. **Merge-preserving** — updates/appends managed API-key lines while leaving comments, blank lines, and unmanaged env vars (notably `LINGTAI_SOUL_FLOW_ENABLED`) intact. Loaded by agents via `env_file` in `init.json`.
- **`tui/internal/config/global.go:451-508`** — `SetEnvVar`: merge-preserving single-key upsert (`value==""` removes) for env vars the TUI owns outside Config.Keys. Used to persist the soul-flow opt-in (`SoulFlowEnabledEnvVar` = `LINGTAI_SOUL_FLOW_ENABLED`, `tui/internal/config/global.go:316`). Companions `SoulFlowEnabled(globalDir)` (`tui/internal/config/global.go:511`) and `SoulFlowEnabledInEnvFile(path)` (`tui/internal/config/global.go:519`) read it back — the former to seed the wizard toggle, the latter for `/kanban`/props to reflect the specific `env_file` an agent points at.
- **`tui/internal/config/global.go:212-218`** — `GlobalDirPath`: PURE path resolver for `~/.lingtai-tui` — resolves `$HOME` and joins `GlobalDirName`, performs no `MkdirAll`, no stat, no write. Added for the no-project launcher's zero-write startup gate (`tui/internal/tui/launcher.go`), which must know where the global dir *would* live without creating it merely by asking.
- **`tui/internal/config/global.go:226-235`** — `EnsureGlobalDir`: the explicit mutating counterpart — resolves via `GlobalDirPath` then `MkdirAll`s it. Callers that want the historical "resolve and create" combined behavior keep using `GlobalDir` (`tui/internal/config/global.go:244-246`), now a one-line wrapper over `EnsureGlobalDir`.
- **`tui/internal/config/global.go:248-309`** — `LoadConfig` / `LoadConfigReadOnly`: both call `loadConfig(dir, tightenPermissions)` for identical read/parse and in-memory legacy-provider-key-name migration behavior. `LoadConfigReadOnly` passes `false` and performs NO filesystem write of any kind. `LoadConfig` passes `true`, preserving the historical permission-tightening order: an existing `config.json` is chmodded to `0600` BEFORE it is read/unmarshaled, including when the later read or parse fails. This ordering is regression-tested by `TestLoadConfigReadOnlyPreservesModeWhileLoadConfigTightensMalformed`. Added because the no-project launcher's draft-purpose `FirstRunModel` must not trigger that chmod before confirmation; every legacy normal/setup/rehydrate construction still reaches `LoadConfig` unchanged.
- **`registry.go`** — preset registry management (see `preset/ANATOMY.md`).
- **`tui/internal/config/registry.go:68-101`** — `RegisteredProject` / `ListRegisteredProjects`: a READ-ONLY registry listing, annotated with a Stat-based `Alive`/`StaleReason` per row but never rewriting `registry.jsonl`, never pruning, never repairing — unlike `LoadAndPrune` (`tui/internal/config/registry.go:103-126`), which remains the explicit prune API called only from normal (already-decided) startup paths. Used by the no-project launcher's "Open Existing" catalog (`tui/internal/tui/launcher.go`) so browsing stopped/registered projects never mutates the registry.

## Connections

- **Called from:** `tui/main.go:151-288`, the manual `lingtai-tui self-update` path, and `tui/internal/tui/firstrun.go:479-554` (`newFirstRunModelForPurpose`'s `config.LoadConfig`/`LoadConfigReadOnly` branch). `NewFirstRunModel` and `NewDraftFirstRunModel` call that body with `purposeNormal` and `purposeDraft` respectively; setup/rehydrate intentionally preserve their legacy behavior by constructing through `NewFirstRunModel` and retagging afterward. `tui/internal/headless/spawn.go:88-173` owns the separate headless bootstrap path. `tui/internal/tui/update.go` (the interactive `/update` command) also calls `InspectKernel`/`RunKernelUpdate` directly for its own confirm-then-update flow, independent of the startup path.
- **Calls out:** PyPI API (`pypi.org/pypi/lingtai/json`), GitHub API (`api.github.com/repos/Lingtai-AI/lingtai/releases/latest`), `uv` / `pip` CLI, and Homebrew for Homebrew-managed TUI binary updates.
- **Bootstrap sequence** (two stages — see `tui/ANATOMY.md` "Shared startup-update preflight" and "Bootstrap sequence" for the full split):
  0. **Preflight, before either stage below and before the no-project decision gate** — `main.go`'s `runVersionPreflight` calls `config.CheckTUIUpgrade` (TUI, read-only) then `config.InspectKernel(globalDir)` (kernel, read-only) unconditionally, for EVERY default interactive launch (existing project, first-run, or the no-project launcher's own picker screen — install.sh installs the kernel by default, so first-run is not exempt). Only when `InspectKernel` reports `NeedsUpdate=true` AND the launch is interactive (real stdin TTY) does `maybeCheckAndPromptKernelUpgrade` prompt — with wording that distinguishes the two actionable cases: `status.Installed==""` (absent, unimportable, or otherwise broken) prints an explicit install/repair prompt (`lingtai kernel is not installed (or is broken). Install/repair it now? [y/N] `), never the generic "update available" phrasing; a present-but-stale kernel prints `lingtai kernel <installed> → <latest>. Update now? [y/N] `. Either way only an explicit "y" calls the mutating `config.RunKernelUpdate(globalDir, force=false)` — this IS the sole automatic install/repair/update opportunity for the whole launch. An already-current/editable/bundle-pinned kernel, a failed lookup, or a decline/EOF/non-interactive launch never mutates. The answer is never persisted or reused across launches.
  1. `config.MigrateLegacyLanguage(globalDir)` then `config.LoadTUIConfig` + `i18n.SetLang` — one-shot language migration and locale resolution, run early (right after the no-project decision gate and `config.GlobalDir()`, i.e. AFTER the preflight above) so startup banners localize; project init/register/utility-skill-refresh happen explicitly, and the rest of this sequence (steps 2-4 below) follows in `tui/main.go`'s `prepareApp`, which itself performs no TUI/kernel version check of its own (that already happened in the preflight) and never installs/repairs the runtime (that also already happened, if consented, in the preflight)
  2. `config.RuntimeReady(globalDir)` — a **read-only** readiness check (never creates/repairs/upgrades); if the runtime still isn't usable at this point (e.g. the preflight's install/repair prompt was declined, or this launch was non-interactive), `prepareApp` surfaces the actionable `*RuntimeReadyError` and refuses to continue rather than silently repairing behind that decision
  3. `preset.Bootstrap(globalDir)` — copy preset resources
  4. `tui.ExportCommandsJSON(globalDir)` — export slash commands
  5. `maybePromptRustToolchain(globalDir)` — one-time optional Rust/Cargo prompt when native file search is unavailable and Cargo is missing

## Notes

- **MCP packages are dependencies of `lingtai`.** `lingtai` on PyPI is a meta-package that bundles `lingtai-kernel` + all addon MCPs. `pip install --upgrade lingtai` upgrades everything. Users never install MCP packages individually.
- **No automatic install/repair path exists outside the preflight.** `config.RuntimeReady` runs on every TUI launch (returning-user `prepareApp`), first-run bootstrap, headless spawn, and the post-Create project-launch phase — but it is READ-ONLY: it never creates, repairs, or upgrades anything, it only reports whether the runtime is usable. The kernel version check AND its consent-gated install/repair/update all happen exactly once, earlier, in `main.go`'s shared `runVersionPreflight` — for EVERY default interactive launch (existing project, first-run, or the empty-directory no-project launcher), not only "returning users," and before any of `prepareApp`'s work. Only an explicit "y" to that preflight's `[y/N]` prompt (real stdin TTY only, per-launch, never persisted) calls the mutating `config.RunKernelUpdate`. If the human declines, or the launch is non-interactive (first-run reached via headless tooling, `spawn`, etc.), the runtime may still be absent/broken when `prepareApp`/the first-run wizard/headless spawn/post-Create launch later check `config.RuntimeReady` — that decline is never silently overridden by a second, hidden install; each of those callers surfaces the actionable `*RuntimeReadyError` instead. The same mutating logic — minus any prompt, since explicit invocation IS the consent — is reachable via `lingtai-tui doctor` / `/doctor` (`RunDoctorUpdate` with `ForceTUI=true` / `ForcePython=true`, surfacing every command's stdout/stderr) and the dedicated `/update` command (`tui/internal/tui/update.go`), which uses `InspectKernel`/`RunKernelUpdate` directly for a kernel-only confirm-then-update flow.
- **Dev mode conversion:** local dev workspaces are enabled explicitly with `$LINGTAI_DEV_ROOT` (for example, `LINGTAI_DEV_ROOT=~/work/GitHub` when that directory contains sibling `lingtai-kernel/` and `lingtai/`). If configured and valid, the runtime should track `lingtai-kernel` editable. `InspectKernel` reports a not-yet-editable dev checkout as `NeedsUpdate=true` (an actionable pending conversion), so it surfaces through the same consent-gated prompt as a PyPI version bump — never automatically; existing editable installs for the same checkout are left alone (`NeedsUpdate=false`) to avoid reinstalling every launch. No common-directory auto-scan runs by default, so ordinary users with source clones are not silently switched off PyPI.
- **`uv` preferred over `pip`:** all pip operations prefer `uv` if available (faster, can auto-download Python).
- **`.env` is no longer clobber-rewritten.** Both `WriteEnvFile` (managed API keys) and `SetEnvVar` (single unmanaged key, e.g. the soul-flow opt-in) preserve everything they don't own: comments, blanks, unrelated vars, and file permissions. This is what lets `LINGTAI_SOUL_FLOW_ENABLED` — written by the first-run/`/setup` wizard opt-in via `preset.GenerateInitJSONWithOpts` — survive a later API-key save. The kernel reads this var from process env (populated from `env_file` at agent boot); the delay/cadence in `manifest.soul.delay` is cadence-only after opt-in, not an on/off switch.
