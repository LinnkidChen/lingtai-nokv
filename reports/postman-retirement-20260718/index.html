PR explainer · 2026-07-18

Retire the dormant UDP Postman path

The Go TUI no longer advertises or runs the cross-machine Postman relay. Remote recipients fail explicitly before any mailbox mutation; local and pseudo-agent filesystem mail remain unchanged.

Decision in one screen

Before
Remote address → write sender outbox → optional Postman UDP/LTPM relay → queue removal without remote acknowledgement.
→
After
Remote address → explicit unsupported error before ID allocation or mailbox writes. Local and pseudo-agent paths continue normally.

Why: Postman was the only in-tree consumer of remote outboxes, had no remote acknowledgement contract, and its CLI/package were otherwise isolated. Removing only the daemon would have stranded newly created queues, so producer behavior changes in the same PR.

Removed

  • lingtai-tui postman dispatch, help, and main function.
  • tui/internal/postman: runner, sender, receiver, wire codec, and tests.
  • Postman-exclusive github.com/klauspost/compress dependency.
  • Uncalled Portal mail writer/cache cluster and its dedicated tests.
  • Deprecated internal SQL-log alias NotificationBlock.
  • Live anatomy/skill claims that Postman remains part of the architecture.

Intentionally retained

  • Local inbox/sent/archive/outbox formats and mailbox IDs.
  • Pseudo-agent polling and same-machine peer/abs delivery.
  • Address parsing, including localhost and IPv6 forms.
  • Portal inbox/archive readers, network topology, and replay.
  • TUI/Portal migration history and parity tests.
  • Existing user outboxes, history, PID files, and running processes.

Independent review

Pool Terra reviewed the frozen candidate read-only. It confirmed the removal/dependency/Portal/alias boundaries and found one end-to-end defect: the TUI ignored WriteMail errors. The final candidate handles that error, preserves retry state and the insight marker, and adds focused coverage.

Terra also corrected platform-split wording and filesystem anatomy citations. The Portal mailbox ID generator remains intentionally retained under the compatibility boundary.

Non-goals and remaining risk

  • No migration, quarantine move, or deletion of existing outbox data.
  • No change to the kernel's filesystem mail contract.
  • No attempt to replace Postman with another network transport.
  • No changes to historical migration sources or old design plans.

An existing remote outbox stays historical/inert. This PR prevents new remote queue creation but does not reinterpret or mutate old records.
