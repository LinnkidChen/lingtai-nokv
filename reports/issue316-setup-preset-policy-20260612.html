Issue #316 — /setup Keep-Current Preset Writes Synthetic Ref

2026-06-12 — worktree fix/setup-preset-policy-316-20260612

Summary

When a user runs /setup and selects Keep current preset (the default cursor position, cursor == -1), the TUI synthesises a placeholder preset with Name="keep_current" to carry the existing agent's LLM/capability config. On save, GenerateInitJSONWithOpts passes that synthetic preset through RefFor(), which derives ~/.lingtai-tui/presets/saved/keep_current.json — a file that does not exist. That path was written to manifest.preset.default and manifest.preset.allowed, which would cause the kernel to reject the config or fail to load the preset at next agent boot.

Root cause

RefFor(p) in tui/internal/preset/preset.go derives a path purely from p.Name and p.Source. It has no way to distinguish real presets from in-memory sentinels. The sentinel is needed so currentPreset() can return the agent's current capabilities to downstream wizard steps, but it should never be serialised to disk. 

// Before fix — GenerateInitJSONWithOpts:
presetRef := RefFor(p)   // "~/.lingtai-tui/presets/saved/keep_current.json" ❌
manifest["preset"] = map[string]interface{}{
    "default": presetRef,  // written!
    "allowed": [..., presetRef],  // written!
}

Fix: three-part patch

PartFileWhat changed
bug 1 preset/preset.go Added isSyntheticPreset(p) helper. In the PreserveActivePreset block that already reads active and allowed from the existing init.json, also read manifest.preset.default. When the passed preset is synthetic, override presetRef with the existing default so keep_current.json is never written.
bug 2 tui/firstrun.go In performSetupSaveOnly(), the propagation call propagatePresetPolicyToNetwork(..., presetCanonicalRef(p), ...) would broadcast keep_current.json to all peer agents. Fixed by reading the real existing default from m.setupKeepInitJSON when m.cursor == -1.
ux 3 tui/firstrun.go In enterAgentPresets(), after the setup-mode hydration block, auto-check the row for the picker cursor's preset if it is not already authorised. This handles the case where the user just created a preset in the editor and it has not yet appeared in the existing allowed list.

Regression test

Added tui/internal/preset/preset_keep_current_test.go with two cases:

  • TestSetupModeKeepCurrentPresetPolicyDoesNotWriteSyntheticPreset — existing init has codex as active/default/allowed; keep-current chosen; user additionally allows zhipu-2; asserts: default=codex, active=codex, allowed={codex,zhipu-2}, no keep_current string anywhere.
  • TestSetupModeKeepCurrentNoExtraAllowed — keep-current chosen, no extra presets granted; asserts existing allowed preserved, default remains codex.

What was NOT changed

  • manifest.preset.allowed remains a static authorization snapshot — this fix does not make it dynamic.
  • The kernel/TUI allowed-gate is unchanged.
  • No new migrations are needed — the corrupt keep_current path would only have appeared in newly-saved init.json files; no migration can retroactively detect and clean it (affected users would simply re-run /setup).
