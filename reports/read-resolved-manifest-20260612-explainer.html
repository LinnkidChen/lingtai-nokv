kernel issue #259 / PR #260 · TUI library · portal fs

Consumers now read system/manifest.resolved.json first

The kernel materializes the active preset in memory; raw init.json is only a creation-time snapshot. Since kernel PR #260 it also publishes the fully-resolved, secret-redacted manifest to <agent>/system/manifest.resolved.json on every boot/refresh. This change makes the TUI library view and the portal manifest reader prefer that artifact, falling back to raw init.json only for stopped / never-booted agents or a malformed artifact.

The artifact

{
  "schema": "lingtai.manifest.resolved/v1",
  "schema_version": 1,
  "generated_at": "2026-06-12T...Z",
  "source": "kernel",
  "manifest": { ...preset-materialized, validated, path-resolved, secret-redacted... }
}

TUI: library skills paths (tui/internal/tui/library.go)

readLibraryPaths previously read manifest.capabilities.skills.paths straight from init.json — so the /library catalog missed paths contributed by the active preset and showed stale paths the preset had dropped. Now:

  • New helper readResolvedLibraryPaths(agentDir) ([]string, bool) reads the artifact. The bool reports whether the artifact was usable.
  • Usable artifact ⇒ authoritative, including when it declares no skills capability (no half-fallback that would resurrect stale init paths).
  • Absent / unreadable / JSON-malformed artifact, missing manifest object, or a skills block that doesn't parse ⇒ fall back to the existing raw init.json logic, unchanged.
  • Downstream behavior (resolveLibraryPath, grouping, catalog build) is untouched; the shared skillsPathsFromCapabilities helper keeps init parsing identical.

Portal: manifest reader (portal/internal/fs/agent.go)

ReadInitManifest is now a two-source read with the same flattening as before:

manifest, err := readResolvedManifest(dir)   // system/manifest.resolved.json
if err != nil {
    manifest, err = readRawInitManifest(dir) // raw init.json fallback
    if err != nil { return nil, err }
}
flattenManifest(manifest)                    // llm.{model,provider,base_url,api_compat,api_key_env}, soul.delay → soul_delay
  • The flattening logic was extracted to flattenManifest and applies identically on both paths — no API or shape change for callers (props panel, system view, network metadata).
  • An artifact whose top-level manifest is missing or not an object counts as malformed and triggers the fallback.
